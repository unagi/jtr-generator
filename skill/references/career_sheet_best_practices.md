# 職務経歴書ベストプラクティスガイド

**対象読者**: LLM（Agent Skills環境）、エンドユーザー
**最終更新**: 2025-12-30
**ステータス**: Phase 1完了

---

## 概要

このガイドは、日本の転職市場で高評価を得るための職務経歴書作成のベストプラクティスをまとめたものです。
人事担当者からのアドバイスとWeb調査に基づき、**再現性のある成果を伝える構造化手法**を提供します。

**TL;DR（LLM用）**:
1. **戦略的カスタマイズ**: 応募企業のJDに合わせて80%以上をカスタマイズ
2. **STAR法の徹底**: 職務経歴はSituation/Task/Action/Resultで構造化し、数値を必ず入れる
3. **PREP法**: 職務要約・自己PRはPoint → Reason → Example → Pointで構造化
4. **逆時系列**: 最新の経験から記載
5. **事実・数値のみ使用**: ユーザーが提示した情報のみを使用し、推測・捏造は厳禁

---

## 1. 全体の原則

### 1.1. 戦略的カスタマイズ

**原則**: 職務経歴書は応募企業のJob Description（JD）に合わせて**80%以上をカスタマイズ**すること。

**理由**:
- 人事担当者は「この人が自社で活躍できるか」を判断する
- 汎用的な職務経歴書では、自社へのフィット感が伝わらない
- 応募企業が求めるスキル・経験を強調することで、書類選考通過率が大幅に向上

**具体例**:
```markdown
❌ 悪い例（汎用的）:
「Pythonでデータ分析を行いました。」

✅ 良い例（カスタマイズ済み）:
「貴社のJDにある『機械学習を活用した顧客分析』に対し、前職では
PythonとTensorFlowを使用した顧客解約予測モデルを構築し、解約率を30%低減しました。」
```

### 1.2. PREP法による構造化

**PREP法**: Point（結論） → Reason（理由） → Example（具体例） → Point（結論の再確認）

**適用箇所**: 職務要約、自己PR

**例**:
```markdown
# 職務要約

**Point（結論）:**
データ分析領域で8年の経験を持ち、前職では機械学習モデル導入により売上予測精度を40%向上。

**Reason（理由）:**
業界知識とPython/AWSスキルを組み合わせ、データドリブン経営を推進してきました。

**Example（具体例）:**
特に、カスタマーサクセス部門では解約率を5% → 2.5%に低減し、MRR 1.2億円の成長に貢献。

**Point（結論）:**
貴社のデータドリブン経営に貢献し、売上成長を加速させます。
```

### 1.3. 逆時系列の徹底

**原則**: 職務経歴は**最新の経験から順に記載**すること。

**理由**:
- 人事担当者が最も知りたいのは「直近の実績とスキル」
- 古い経験から記載すると、最新の状態が伝わりにくい
- 逆時系列は日本の転職市場で標準的なフォーマット

**推奨順序**:
1. 職務要約（3-5行）
2. 職務経歴（**最新 → 古い順**）
3. 活かせる経験・スキル
4. 自己PR

### 1.4. 履歴書の時系列整合（LLM向けの具体ルール）

**目的**: 面接に呼ぶ理由が短時間で腹落ちするよう、履歴書の一貫性を担保する。

**ルール**:
- 職歴は原則「入社 → 退職」のペアで記載する
- 最終行は状態を明記する（現職なら「現在に至る」、退職後で所属なしならその旨）
- 入社/退職の片方が欠ける場合は生成前に確認する

**記載例**:
```yaml
work_history:
  - date: 2022-03
    type: 退職
    company: 株式会社〇〇
    note: 退職後、現在所属なし（求職中）
```

## 2. STAR法の徹底

### 2.1. STAR法とは

**STAR法**: Situation（状況） → Task（課題） → Action（行動） → Result（成果）

**目的**: 再現性のある成果を構造的に伝える

**適用箇所**: 職務経歴の各エピソード

### 2.2. 各要素の書き方

#### Situation（状況）

**定義**: エピソードの背景・文脈を説明

**ポイント**:
- 会社の規模、業種、部署の役割を簡潔に記載
- 問題が発生した背景を明確にする

**例**:
```markdown
**Situation:**
創業5年のSaaS企業で、カスタマーサクセス部門の立ち上げを担当。
顧客解約率が月5%と高く、収益成長の阻害要因となっていた。
```

#### Task（課題）

**定義**: 自分が解決すべき具体的な課題

**ポイント**:
- 数値目標を明示する
- 期限を含める（3ヶ月で、年内に、等）

**例**:
```markdown
**Task:**
解約率を3ヶ月で3%以下に低減し、年間MRR 1億円の維持を達成する。
```

#### Action（行動）

**定義**: 自分が取った具体的な行動

**ポイント**:
- **自分の貢献を明示**（チームの成果ではなく、「私が主導」「私が実施」等）
- 複数の行動がある場合は箇条書きで整理
- 技術スタックを明記（Python, AWS, Salesforce等）

**例**:
```markdown
**Action:**
- 顧客インタビュー20社実施し、解約理由を分析（オンボーディング不足が60%）
- オンボーディングプログラムを再設計（初月導入支援を強化）
- チャーンリスク予測モデルを構築（Python + Salesforceデータ連携）
```

#### Result（成果）

**定義**: 行動の結果として得られた成果

**ポイント**:
- **必ず数字を入れる**（売上↑30%、コスト↓20%、解約率↓50%等）
- 目標達成の有無を明記
- 副次的な成果も含める（NPS向上、表彰、社内標準化等）

**例**:
```markdown
**Result:**
- 解約率を5% → 2.5%に低減（目標達成）
- MRR 1.2億円に成長（20%増）
- Net Promoter Score（NPS）を30 → 55に改善
```

### 2.3. STAR法の完全な例

```markdown
### 株式会社○○（2020-04 〜 2023-03）
カスタマーサクセス部門 マネージャー

**業種**: SaaS（従業員50名）

#### 担当業務と実績

**Situation（状況）:**
創業5年のSaaS企業で、カスタマーサクセス部門の立ち上げを担当。
顧客解約率が月5%と高く、収益成長の阻害要因となっていた。

**Task（課題）:**
解約率を3ヶ月で3%以下に低減し、年間MRR 1億円の維持を達成する。

**Action（行動）:**
- 顧客インタビュー20社実施し、解約理由を分析（オンボーディング不足が60%）
- オンボーディングプログラムを再設計（初月導入支援を強化）
- チャーンリスク予測モデルを構築（Python + Salesforceデータ連携）

**Result（成果）:**
- 解約率を5% → 2.5%に低減（目標達成）
- MRR 1.2億円に成長（20%増）
- Net Promoter Score（NPS）を30 → 55に改善
```

## 3. 推奨される項目と並び順

### 3.1. 標準的な職務経歴書の構成

プロフィール/免許・資格は履歴書データから自動生成されるため、Markdown本文には記載しません。

```markdown
## 職務要約
（3-5行、PREP法適用）

## 職務経歴
（最新 → 古い順、STAR法適用）

### 株式会社○○（2020-04 〜 2023-03）
職種・役職

**業種**: SaaS（従業員50名）

#### 担当業務と実績
（STAR法で記述）

### 株式会社△△（2015-04 〜 2020-03）
職種・役職

**業種**: EC（従業員300名）

#### 担当業務と実績
（箇条書きまたはSTAR法）

## 活かせる経験・スキル

### 技術スキル
- **Python（5年）**: 機械学習モデル構築（scikit-learn, TensorFlow）
- **AWS（3年）**: EC2, S3, Lambda活用

### ソフトスキル
- **チームマネジメント**: 5名チームのリーダーとして年間目標120%達成
- **問題解決力**: STAR法で構造化した課題解決（上記実績参照）

## 自己PR
（PREP法適用、200-300文字）
```

### 3.2. 各項目の記載ポイント

#### 職務要約

**目的**: 3-5行で「自分はどんな人材か」を要約

**記載内容**:
- 専門領域と経験年数
- 代表的な実績（数字1つ）
- 応募企業への貢献（カスタマイズ必須）

**例**:
```markdown
# 職務要約

データ分析領域で8年の経験を持ち、前職では機械学習モデル導入により売上予測精度を40%向上。
業界知識とPython/AWSスキルを活かし、貴社のデータドリブン経営に貢献いたします。
```

#### 職務経歴

**目的**: 各職場での経験と成果を時系列で説明

**記載内容**:
- 会社名、在籍期間（YYYY-MM形式）
- 職種・役職
- 業種と会社規模
- STAR法による実績記述

**詳細度の調整**:
- 直近1-2社: 詳細に記述（STAR法適用）
- それ以前: 箇条書きで簡潔に

#### 活かせる経験・スキル

**目的**: 技術スキルとソフトスキルを整理

**記載内容**:
- 技術スキル: 言語・ツール・経験年数
- ソフトスキル: マネジメント、問題解決力、コミュニケーション等

**ポイント**:
- 応募企業のJDに合わせて優先順位を調整
- **プロフィール/免許・資格は履歴書データから自動生成**されるため、Markdown本文には記載不要

#### 自己PR

**目的**: 自分のキャリアの軸と応募企業への貢献を要約

**記載内容**:
- PREP法で構造化（200-300文字）
- 応募企業への具体的な貢献内容

**例**:
```markdown
## 自己PR

私のキャリアの軸は「データを活用した意思決定支援」です。
前職では解約率低減プロジェクトを通じて、データ分析→施策立案→実行→検証のサイクルを確立しました。
この経験を活かし、貴社のデータドリブン経営に貢献し、売上成長を加速させます。
```

## 4. 優秀者向けベストプラクティスTips

### 4.1. 数字とエピソードで「再現性」を証明

**問題**: 優秀な人ほど「やったことを淡々と書く」傾向がある

**解決策**:
- 数字を必ず入れる（売上↑、コスト↓、工数↓、解約率↓等）
- エピソードで「どう考えて行動したか」を明示
- 再現性を証明する（「同じ状況で再度成果を出せるか」）

**例**:
```markdown
❌ 悪い例:
「顧客解約率を改善しました。」

✅ 良い例:
「顧客インタビュー20社を実施し、解約理由の60%がオンボーディング不足と特定。
オンボーディングプログラムを再設計した結果、解約率を5% → 2.5%に低減しました。」
```

### 4.2. 逆算で詳細度を調整（「ここだけは詳細に」）

**問題**: すべてを詳細に書こうとすると、完璧主義に陥る

**解決策**:
- 応募企業のJDに最も関連する実績を1-2つ選ぶ
- その実績のみSTAR法で詳細に記述
- それ以外は箇条書きで簡潔に

**例**:
```markdown
### 株式会社○○（2020-04 〜 2023-03）
カスタマーサクセス部門 マネージャー

#### 担当業務と実績（詳細）

**Situation/Task/Action/Result:**
（STAR法で詳細に記述）

### 株式会社△△（2015-04 〜 2020-03）
データアナリスト

#### 担当業務（簡潔）
- 売上予測モデルの構築（Python/scikit-learn）
- ダッシュボード開発（Tableau）による経営KPI可視化
- A/Bテスト設計・分析（コンバージョン率15%向上）
```

### 4.3. 完璧主義を避け、合格点を目指す

**問題**: 優秀な人ほど「完璧な職務経歴書」を目指し、時間をかけすぎる

**解決策**:
- 職務経歴書は「合格点」を取れば十分（面接で詳細を説明できる）
- 80%の完成度で応募し、フィードバックを得てから改善
- 時間をかけすぎず、複数社に応募してPDCAを回す

**目安**:
- 初回作成: 2-3時間
- カスタマイズ: 応募企業ごとに30分-1時間

## 5. よくある失敗パターンと対策

### 5.1. 業務内容の羅列

**失敗パターン**:
```markdown
❌ 悪い例:
「データ分析を担当しました。Pythonを使用しました。レポートを作成しました。」
```

**対策**: STAR法で成果を明示
```markdown
✅ 良い例:
「売上予測モデルを構築（Python/scikit-learn）し、予測精度を40%向上。
その結果、在庫最適化により年間2000万円のコスト削減を実現しました。」
```

### 5.2. 時系列が古い順

**失敗パターン**:
```markdown
❌ 悪い例:
## 職務経歴

### 株式会社△△（2015-04 〜 2020-03）
（古い経験から記載）

### 株式会社○○（2020-04 〜 2023-03）
（最新の経験が後ろ）
```

**対策**: 逆時系列へ変更
```markdown
✅ 良い例:
## 職務経歴

### 株式会社○○（2020-04 〜 2023-03）
（最新の経験を最初に記載）

### 株式会社△△（2015-04 〜 2020-03）
（古い経験は後ろ）
```

### 5.3. 数値が抽象的

**失敗パターン**:
```markdown
❌ 悪い例:
「解約率を大幅に改善しました。」
「売上に貢献しました。」
```

**対策**: 具体的な数値に変換
```markdown
✅ 良い例:
「解約率を5% → 2.5%に低減（50%改善）」
「売上を前年比30%増（1億円 → 1.3億円）に成長」
```

### 5.4. 主語が不明確

**失敗パターン**:
```markdown
❌ 悪い例:
「チームで解約率改善プロジェクトを実施しました。」
```

**対策**: 主語を「私」に変更し、自分の役割を明示
```markdown
✅ 良い例:
「私が主導し、顧客インタビュー20社を実施。チームで解約理由を分析し、
オンボーディングプログラムを再設計しました。」
```

### 5.5. 汎用的すぎる内容

**失敗パターン**:
```markdown
❌ 悪い例:
「Pythonでデータ分析を行いました。」
```

**対策**: 応募企業のJDに合わせてカスタマイズ
```markdown
✅ 良い例（応募企業がEC企業の場合）:
「貴社のJDにある『レコメンドエンジンの構築』に対し、前職ではPythonと協調フィルタリングを使用した
レコメンドシステムを構築し、クリック率を25%向上させました。」
```

## 6. SDS法（補足）

### 6.1. SDS法とは

**SDS法**: Summary（要約） → Details（詳細） → Summary（要約）

**適用箇所**: 職務要約、自己PR（PREP法との選択）

**特徴**:
- PREP法よりも簡潔
- 詳細を挟むことで説得力を持たせる

**例**:
```markdown
**Summary（要約）:**
データ分析領域で8年の経験を持ち、機械学習モデル導入による業績改善を実現。

**Details（詳細）:**
前職では顧客解約予測モデルを構築し、解約率を5% → 2.5%に低減。
Python/TensorFlow/AWSを活用したデータドリブン経営を推進しました。

**Summary（要約）:**
この経験を活かし、貴社の売上成長に貢献いたします。
```

## 7. Markdown活用のポイント

### 7.1. 推奨されるMarkdown構文

職務経歴書では以下のMarkdown構文を活用すると、視認性が向上します：

#### 見出し

```markdown
## 職務要約（H2: トップレベル）
## 職務経歴（H2）
### 株式会社○○（H3）
#### 担当業務と実績（H4）
```

#### 箇条書き

```markdown
**Action:**
- 顧客インタビュー20社実施
- オンボーディングプログラム再設計
- チャーンリスク予測モデル構築
```

#### 太字

```markdown
**Situation:** 創業5年のSaaS企業で...
**Task:** 解約率を3ヶ月で3%以下に低減...
**Action:** 顧客インタビュー20社実施し...
**Result:** 解約率を5% → 2.5%に低減...
```

### 7.2. Markdown記述時の注意

1. **空行で段落を区切る**: 改行だけでは段落が分かれない
2. **見出しレベルの統一**: H2 → H3 → H4の階層を守る（H1は使用しない）
3. **箇条書きの階層**: インデントで子項目を表現（将来拡張）

## 8. Agent Skills環境でのサポート

### 8.1. LLMの役割

Agent Skills環境では、LLMが以下のサポートを提供します：

1. **ベストプラクティスの提示**
   - このガイドの内容をユーザーに説明
   - PREP法、STAR法の具体例を提示

2. **対話的な構造化支援**
   - Situation/Task/Action/Resultを段階的に聞き出す
   - ユーザーの回答を整理し、Markdown形式で構造化

3. **数値化の徹底**
   - 曖昧な表現（「大幅に改善」等）を数値に変換するよう促す
   - Before/Afterの数値を明示させる

4. **カスタマイズ支援**
   - 応募企業のJDを分析し、強調すべきスキル・経験を提案
   - JDとの整合性をチェック

5. **事実・数値の取り扱い（重要）**
   - **ユーザーが提示した事実・数値のみを使用**（推測・捏造は厳禁）
   - 不明点は必ず質問し、確定しない情報は「要確認」と明記
   - 数値がない場合は、ユーザーに聞き出すか定性的表現に留める

6. **Markdown構文の制約（重要）**
   - **解析基準**: GitHub Flavored Markdown（GFM）
   - **対応構文**: 見出し（##/###/####）、箇条書き、タスクリスト、太字/斜体、取り消し線、リンク、水平線、コード、表
   - 最終出力はGFMの範囲に収めること

### 8.2. 対話フロー例

```
LLM: 職務経歴書を作成しますね。まず、応募企業の情報を教えていただけますか？
     （例: 企業名、職種、Job Description）

User: 株式会社サンプル、データアナリスト、機械学習・AWS経験者歓迎

LLM: ありがとうございます。次に、直近の職務経験について教えてください。
     まず「Situation（状況）」から伺います。どのような会社で、どのような課題がありましたか？

User: SaaS企業で、顧客解約率が高かった

LLM: もう少し具体的に伺えますか？会社の規模、解約率の数値、なぜ問題だったかを教えてください。

User: 従業員50名、解約率5%、収益成長の阻害要因

LLM: わかりました。次に「Task（課題）」を伺います。あなたが解決すべき具体的な目標は何でしたか？

（以下、同様に対話を続け、STAR法で構造化）
```

## 10. まとめ

### 10.1. 重要ポイントの再確認

1. **戦略的カスタマイズ**: 応募企業のJDに合わせて80%以上をカスタマイズ
2. **STAR法の徹底**: Situation/Task/Action/Resultで構造化し、数値を必ず入れる
3. **逆時系列**: 最新の経験から記載
4. **PREP法**: 職務要約・自己PRはPoint → Reason → Example → Pointで構造化
5. **完璧主義を避ける**: 合格点を目指し、PDCAを回す

### 10.2. チェックリスト

職務経歴書作成時は以下を確認してください：

- [ ] 応募企業のJDを分析し、80%以上をカスタマイズしたか？
- [ ] 職務経歴はすべてSTAR法で構造化したか？
- [ ] Resultに数値を入れたか？（売上↑、コスト↓、解約率↓等）
- [ ] 時系列は逆順（最新 → 古い順）になっているか？
- [ ] 職務要約・自己PRはPREP法またはSDS法で構造化したか？
- [ ] 主語が明確か？（チームではなく「私」の貢献を明示）
- [ ] 曖昧な表現（「大幅に改善」等）を排除し、具体的な数値に変換したか？
